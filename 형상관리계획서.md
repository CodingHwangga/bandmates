## 3. 버전 규칙

**형식**: v주.부.수

예시:
- v1.0.0 → 요구사항 변경
- v1.1.0 → 기능 추가
- v1.1.1 → 오타 수정

---

## 5. 브랜치 전략

- **main**: 최종 승인된 버전
- **draft**: 작업 중
- **review**: 검토 대기
- **experiment**: 실험용

---

## 6. Commit 규칙

**형식**: `[구분자] 변경 내용`

예시:
```
[FEAT] 밴드 생성 기능 추가
[FIX] 일정 버그 수정
[DOCS] 요구사항서 작성
[CR-01] 파일 업로드 용량 변경
```

**주 1회 이상 commit 필수**

---

## 7. 변경 관리

변경 필요 시:
1. 변경 이유 확인
2. 별도 브랜치에서 수정
3. 검토 후 main에 병합
4. CHANGELOG.md에 기록
